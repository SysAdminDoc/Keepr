# Project Research and Feature Plan

> Research date: 2026-05-25 · Target HEAD: `d67484e` (initial v0.1.0 release) · Repo: https://github.com/SysAdminDoc/Keepr

## Executive Summary

Keepr v0.1.0 is a clean, well-shaped Tauri 2 + React + Rust + SQLite scaffold of a pixel-close Google Keep clone — masonry grid, pinned/others split, 12 Keep colors, text and checklist notes, labels, archive/trash, light/dark theme, manual ZIP backup. The product identity is the right one: stay a Keep-aesthetic, single-user, offline-first card app and resist any pull toward outliner/wiki/Notion territory. The current code is small enough to iterate cleanly but ships with a concentrated set of **P0 data-loss and security risks** in the ZIP import/export pipeline and the editor, an essentially absent accessibility layer, no test or CI surface, and a doc/config drift that contradicts the "portable EXE" pitch. The highest-value direction for v0.2 is **earn trust before adding features**: close the audit's P0 list, add a tiny test+CI surface, ship a real portable bundle, and lay the data primitives (attachments protocol, schema migration, FTS) that the v0.3 multimodal/reminders work needs. Only after that should the project chase the feature-parity gaps with real Keep.

**Top 10 opportunities, priority order:**

1. **P0 — Fix the ZIP import/export pipeline** (zip-slip, WAL pages lost on export, in-memory-throwaway stranded on partial import failure, no schema-version table). Today a malformed backup file is an RCE vector and a normal restore can silently destroy data. ([audit-code §1, items 1–5](.audit-notes-code.md))
2. **P0 — Fix the editor's draft-clobber and re-entrant close** (`existing` ref churn wipes typing; backdrop close is re-entrant; ALT-F4 silently discards drafts). Users will lose work the first time two edits race. ([audit-code §3, items 42–44](.audit-notes-code.md))
3. **P0 — Tighten the Tauri capability surface** (`csp: null` + unused broad `fs:*`). The renderer doesn't use the fs plugin at all; the open CSP turns any future inline-HTML escape into RCE. ([audit-code §2, items 23–24](.audit-notes-code.md))
4. **P0 — N+1 in `list_notes`** (3 queries per note × every save). At 500 notes the UI gets visibly worse on every pin/archive. Single 3-query batched load is a half-day fix. ([audit-code §6, item 99](.audit-notes-code.md))
5. **P1 — Accessibility pass** (no aria-labels, no focus trap, no escape on Settings/Labels, no aria-live for toasts, no keyboard shortcuts). The base is a small enough surface to fix in one pass. ([audit-code §5](.audit-notes-code.md))
6. **P1 — Tests + CI scaffold** (zero tests, no `.github/workflows`). Pick three Rust integration tests around `import_zip` and `set_trashed`, plus a vitest pass over `filterNotes`/store actions, plus a `tauri-driver` smoke test that opens a note. ([audit-code §9, items 135–137](.audit-notes-code.md))
7. **P1 — Real portable EXE bundle target + portable-mode detection** (README claims "portable EXE" but `bundle.targets` is `["nsis","msi"]`; data dir is fixed to `%APPDATA%\com.sysadmindoc.keepr\`, README claims `%APPDATA%\Keepr\`). ([audit-code §2/7/10, items 26–27, 105, 138–139](.audit-notes-code.md))
8. **P1 — Image attachments via custom `keepr-resource://` protocol** (Joplin's `:/resource-id` pattern). This is both the #1 user-missed Keep feature and a primitive that v0.3 drawings, OCR, and Takeout import all depend on. ([keep §1](.audit-notes-keep.md), [competitors §2 + "5 patterns" §4](.audit-notes-competitors.md))
9. **P1 — Reminders v1** (time-based, with native Windows toast via `tauri-plugin-notification`). Second-most-used Keep button after the checkbox; without it Keepr is "sticky notes." Skip location reminders (Google deprecated them in late 2025). ([keep §2](.audit-notes-keep.md))
10. **P1 — Keyboard shortcuts + help overlay** (Keep's canonical set: `c`/`l`/`/`/`?`/`j`/`k`/`e`/`f`/`#`/`x`/`Ctrl+G`). Free power-user appeal; one-day implementation. ([keep §9](.audit-notes-keep.md))

## Current Product Map

### Core workflows (v0.1)
1. **Capture a quick note** — click "Take a note…" → editor modal opens → type → click outside → save
2. **Capture a checklist** — same path, click the list icon in `NewNoteBar` or toggle the list-mode button in the editor
3. **Organize an existing note** — click a card → editor opens → edit title/body/color/labels/pin → click outside → save
4. **Triage** — hover card → click archive icon (move out of main view) or trash icon (move to Trash)
5. **Restore** — go to Trash → click restore on a card, or Empty Trash via the button
6. **Find a note** — type in the top-bar search; results filter by title/body/checklist match
7. **Filter by label** — click a label row in the sidebar; grid filters to that label
8. **Manage labels** — sidebar "Edit labels" → create/rename/delete
9. **Backup** — Settings → Export backup… → save ZIP to chosen path
10. **Restore from backup** — Settings → Restore backup… → pick ZIP, confirm

### Existing features (high level — full inventory below)
- Pixel-close Keep UI: masonry grid, pinned/others sections, color palette, label chips, sidebar navigation
- Two note kinds: text and checklist
- Sections: Notes, per-Label, Archive, Trash
- Color picker (12 Keep colors, light + dark variants)
- Label CRUD
- Search across title + body + checklist text
- Pin / archive / trash / restore / delete forever
- Empty Trash
- Light / dark theme toggle (localStorage persisted)
- Manual ZIP export + ZIP import via Tauri dialog
- Refresh button

### User personas
1. **Offline-first private user** — the explicit target. Wants a Google Keep that works on a laptop with no internet, with their notes on their own disk. Backs up to a synced cloud folder (Drive, OneDrive) on their own schedule. *Primary persona — every priority decision in this doc favors them.*
2. **Migrating Keep user** — has years of Keep notes (text, lists, images, drawings, reminders) and wants out of Google's ecosystem without losing everything. Will use Google Takeout to export. *Will become primary once Takeout import lands (P1).*
3. **Power keyboard user** — wants `c` for compose, `/` for search, `j`/`k` to navigate, `?` for help. *Currently unaddressed.*
4. **Sensitive-info user** — keeps banking notes, recovery codes, or other secrets in Keep. Wants per-note or whole-app lock. *Currently unaddressed; competitive opportunity vs Keep, which has no lock at all.*
5. **Sticky-note user** — wants a tray icon, quick-capture popup, "show all / hide all" gesture. *Currently unaddressed.*

### Platforms and distribution channels
- **Windows desktop** is the only platform currently configured (`bundle.targets = ["nsis", "msi"]`)
- **macOS / Linux** — Tauri supports them but no icons, no signing, no docs
- **Mobile** — Tauri Mobile exists but the data model and UI density assume desktop. Out of scope for v0.x
- **Distribution** — public GitHub repo at https://github.com/SysAdminDoc/Keepr; no release built yet, no `gh release create`, no signed binaries

### Important integrations / permissions / storage / data flows
- **Storage:** SQLite at `app_data_dir()/keepr.db` (= `%APPDATA%\com.sysadmindoc.keepr\keepr.db` on Windows). Schema: `notes`, `checklist_items`, `labels`, `note_labels` (4 tables, foreign keys ON, WAL mode ON, no version pragma, no FTS).
- **Tauri plugins enabled:** `tauri-plugin-dialog` (used), `tauri-plugin-fs` (declared but unused, see audit item 24).
- **Capabilities granted:** `core:default`, `dialog:default`, `dialog:allow-open`, `dialog:allow-save`, `fs:default`, `fs:allow-read-file`, `fs:allow-write-file`.
- **Network:** none. The app makes zero outbound requests at runtime; intentional for the offline-first identity.
- **Data flow:** all UI state lives in a single zustand store; mutations call Tauri commands which write SQLite; `load()` re-fetches the full notes+labels list and replaces the store atomically. No optimistic updates.

## Competitive and Ecosystem Research

Compressed from `.audit-notes-competitors.md`. For each app: notable capabilities, what to learn, what to avoid.

### anselm94/googlekeepclone (closest visual reference)
- **Learn:** Material elevation + Roboto pairing nails the Keep look; theme-aware paired color palette confirms Keepr's design choice.
- **Avoid:** server-side webapp architecture for a single-user offline app. Keepr's Tauri+SQLite is the correct call.

### Joplin (offline-first + 7 sync backends + plugins)
- **Learn:**
  - `:/resource-id` URI scheme for attachments — adopt as `keepr-resource://<id>`. ([keepr-resource pattern](.audit-notes-competitors.md))
  - Open-Markdown export with per-note `.md` files and YAML frontmatter — longevity insurance.
  - "Local filesystem" sync driver = let the user's existing Drive/OneDrive/Syncthing folder do the sync work.
  - Honest conflict UX (Conflicts notebook); the more humane variant is a per-card conflict banner with side-by-side diff.
- **Avoid:** Joplin's plugin marketplace is a security liability; if Keepr ever adds plugins, sandbox aggressively via Tauri's `isolation` pattern.

### Notesnook (E2EE + vault + app lock)
- **Learn:** two-tier security model — App Lock (PIN/biometric, locks entire DB after N min inactivity) + Private Vault (separate password, gates specific notes). Argon2id + XChaCha20-Poly1305 primitives. Broad importer matrix (Evernote / Keep / Joplin / Obsidian / HTML / MD).
- **Avoid:** forced-account-creation onboarding; feature-paywall fragmentation. Keepr's instant-open + MIT model is strictly better.

### Memos (single-binary microblog)
- **Learn:** inline `#hashtag` tag input (vastly better than menu-based labeling); single-binary distribution model.
- **Avoid:** append-only chronological timeline (wrong for Keep's pin+masonry paradigm).

### AppFlowy (Notion-class workspace)
- **Learn:** self-hostable sync server as a *separate* SKU from the app itself; app stays installable without "create account" friction.
- **Avoid:** Notion-clone scope (database views, AI block widgets, block editing). Anti-Keep.

### Blinko (Tauri + AI-RAG)
- **Learn:** explicit macOS Gatekeeper guidance for the quarantine attribute in the README (Keepr will need the same if macOS builds ship).
- **Avoid:** PostgreSQL backend (overkill for single-user); 92% TS / 2.9% Rust ratio (move logic to Rust, justify Tauri).

### Trilium / Standard Notes / Obsidian / Logseq (hierarchical / extension / vault / outliner)
- **Learn:**
  - Standard Notes' sandboxed-iframe extension model if Keepr ever ships plugins.
  - Obsidian's "vault = folder of .md" as a complementary export format.
  - Logseq's daily-journal pattern as inspiration for a future "today's note" quick action.
- **Avoid:** hierarchical folders (Trilium), user-scripts (Trilium), feature paywall (Standard Notes Plus), closed source (Obsidian), outliner paradigm (Logseq) — every one of these breaks Keep's identity.

### Built-in OS sticky notes (Windows 11, macOS Notes, KDE Marknote)
- **Learn:** Windows 11's tray icon with "Show all / Hide all" gesture; Marknote's store-as-Markdown-files option.
- **Avoid:** macOS Notes' restrictions on locked tagged notes (orthogonality matters).

### Cross-cutting themes
1. **Local-filesystem sync driver pattern wins** across Joplin/Obsidian/Logseq/Marknote. Keepr should adopt "write to a folder you point at your existing cloud sync" as the next sync model, not build a sync server.
2. **E2EE-at-rest for sync** is universal hygiene in 2026 — adopt when sync ships.
3. **Open / extractable format** is the longevity story. Keepr's SQLite + ZIP is good; a `.md` vault export mode seals it.
4. **Table-stakes 2026 features Keepr is missing:** Markdown formatting, image attachments, inline `#hashtag` input, drag-reorder, reminders, app lock/vault, import from other apps, plain-text/Markdown export.

## Highest-Value New Features

### NF-01 — Image attachments (multi-image per note)
- **Problem solved:** Keep is fundamentally multimodal; text-only is "kinda like Keep."
- **Evidence:** [keep §1](.audit-notes-keep.md); [competitors cross-cutting "table-stakes"](.audit-notes-competitors.md).
- **Behavior:** drop / paste / pick image → stored on disk under `app_data_dir()/resources/<uuid>.<ext>` → embedded in note body via `keepr-resource://<uuid>` custom protocol → rendered above title/body in the card; 1/2/3/4+ images use Keep's grid breakpoints; up to ~20 per note.
- **Implementation:** new Tauri custom protocol handler (`src-tauri/src/main.rs` or a dedicated module); new `attachments` table `(id, note_id, kind, mime, byte_size, created_at)`; React `<img>` and drop-zone in `NoteEditor`; per-card image-grid in `NoteCard`.
- **Data / API:** `add_attachment(note_id, file_path)`, `delete_attachment(id)`. ZIP export bundles `resources/<id>.<ext>`; ZIP import re-registers. Markdown-vault export (NF-08) rewrites to `_resources/<filename>` relative paths.
- **Risks:** attachment dedup, orphan cleanup on `empty_trash`, EXIF stripping for privacy, large-image memory (consider on-demand resize for the card thumb).
- **Verification:** integration test that creates a note + attachment, exports ZIP, imports ZIP into a fresh data dir, asserts the image renders. Manual: drop a 5 MB JPEG into a note; check render time and DB file size growth.
- **Complexity:** L. **Priority:** P1.

### NF-02 — Reminders v1 (time-based + recurring + sidebar section)
- **Problem solved:** Keep's reminder bell is the second-most-used button; without it Keepr is sticky notes.
- **Evidence:** [keep §2](.audit-notes-keep.md); Google deprecated *location* reminders in late 2025 so we **skip those**.
- **Behavior:** bell icon in editor → "Later today / Tomorrow / Next week / Pick…" → optional repeat (RRULE-compatible). Sidebar gets a Reminders section sorting by next-fire-time, overdue items at top in a red-tinted tag. Native Windows toast at fire time via `tauri-plugin-notification`.
- **Implementation:** new `reminders` table `(id, note_id, fire_at, rrule, snooze_until, fired_at, dismissed_at)`; Rust scheduler thread that polls `WHERE fire_at <= now AND fired_at IS NULL`; bridge to `tauri-plugin-notification`; "Snooze 10 min / 1 h / Tomorrow morning" actions on toast (Windows 10/11 supports notification actions).
- **Data / API:** `set_reminder`, `clear_reminder`, `snooze_reminder`. Stored as RFC 5545 RRULE strings for future Calendar/ICS export.
- **Risks:** OS notification permission on Win 11; scheduler running while app is in tray vs closed (toast in background only works while app process is alive — document this); DST/timezone handling around RRULE.
- **Verification:** integration test that schedules a reminder 2s in the future and asserts the scheduler ticks. Manual: schedule a reminder, close to tray, observe toast fires.
- **Complexity:** L. **Priority:** P1.

### NF-03 — Keyboard shortcuts + help overlay
- **Problem solved:** power-user appeal; Keep parity. Currently zero shortcuts.
- **Evidence:** [keep §9](.audit-notes-keep.md) — canonical list verified from [Google docs](https://support.google.com/keep/answer/12862970).

## Existing Feature Improvements

Each item has a P0/P1/P2/P3 priority. Cross-reference to `.audit-notes-code.md` items in brackets.

### EI-01 [P0] Fix zip-slip in `import_zip` ([item 1](.audit-notes-code.md))
- **Current:** loop unconditionally writes `staging.join(p)`; no validation of `p.components()`; no entry-count cap; no decompressed-size cap.
- **Recommended:** reject entries with `ParentDir`/`RootDir`/`Prefix` components; cap entry count (e.g. ≤ 10 000); cap total uncompressed bytes (e.g. ≤ 2 GB or 10× the input ZIP's size).
- **Touches:** `src-tauri/src/commands.rs:485-500`.
- **Backcompat:** none — only rejects malicious inputs.
- **Verify:** Rust integration test with a crafted zip containing `../escape.txt`.

### EI-02 [P0] WAL checkpoint before `export_zip` ([item 4](.audit-notes-code.md))
- **Current:** `export_zip` skips `-wal`/`-shm`/`-journal` (good) but never checkpoints, so committed WAL pages silently miss the backup.
- **Recommended:** `conn.execute_batch("PRAGMA wal_checkpoint(TRUNCATE)")` under the mutex before opening the file for write.
- **Touches:** `src-tauri/src/commands.rs:437-464`.
- **Verify:** test: open conn → insert row → export ZIP without restarting → re-import into a fresh data dir → assert the row is present.

### EI-03 [P0] Safe-swap on `import_zip` partial failure ([items 2–3](.audit-notes-code.md))
- **Current:** if `std::fs::copy` or `crate::db::open` fails after the throwaway in-memory swap, the throwaway stays as the live conn → silent data loss until restart.
- **Recommended:** rename current `keepr.db` → `keepr.db.prev` before copying restored DB over; on any failure after the swap, restore from `.prev` and reopen; only delete `.prev` after the new open succeeds. Also add a `busy: AtomicBool` gate so renderer can't fire other commands during import.
- **Touches:** `src-tauri/src/commands.rs:466-528`.
- **Verify:** test: inject a failure mid-swap; assert `.prev` is restored and conn is valid.

### EI-04 [P0] Add `PRAGMA user_version` + migration framework ([item 5](.audit-notes-code.md))
- **Current:** `CREATE TABLE IF NOT EXISTS` is the entire migration story. v0.2 schema additions will silently fail on v0.1 databases.
- **Recommended:** add `migrate(conn)` in `src-tauri/src/db.rs` that reads `PRAGMA user_version`, runs ordered up-migrations, and bumps the pragma transactionally. Set v0.1 baseline to `user_version = 1` retroactively (idempotent for fresh installs).
- **Touches:** `src-tauri/src/db.rs`.
- **Verify:** test: open a v0.1-like DB (no `attachments` table); call `migrate`; assert `attachments` table now exists at `user_version = 2`.

### EI-05 [P0] Fix CSP and tighten capabilities ([items 23–24](.audit-notes-code.md))

## Reliability, Security, Privacy, and Data Safety

### Bugs / risks found
- **Data-loss class:** zip-slip on import (EI-01); WAL pages lost on export (EI-02); in-memory throwaway stranded on import partial failure (EI-03); no schema migration (EI-04).
- **Crash class:** `app_data_dir().expect()` panics if APPDATA is misconfigured (audit item 25).
- **Lock contention:** mutex held through `tx.commit()` + re-reads in `create_note`/`update_note` (EI-26).
- **Editor races:** `existing`-ref clobber (EI-06); re-entrant close handler + ALT-F4 discard (EI-07).
- **State leakage:** `set_pinned` silently un-archives ([item 6](.audit-notes-code.md)); `set_trashed` does not preserve prior archive state ([item 8](.audit-notes-code.md)); `delete_note_permanent` not gated to trashed-only ([item 10](.audit-notes-code.md)).
- **Lossy round-trip:** list↔text kind change drops `checked` state (EI-22).

### Missing guardrails
- No `WHERE trashed=0` on `set_pinned`/`set_color`/etc. ([item 7](.audit-notes-code.md)).
- No CHECK constraints on `notes.kind` or `notes.color` ([item 15](.audit-notes-code.md)).
- No input length caps anywhere (EI-33).
- No exclusive "import in progress" gate (EI-03).

### Permission / network / file-system concerns
- `csp: null` (EI-05) is the largest single risk: any future XSS becomes RCE through Tauri commands.
- Unused broad `fs:*` capabilities (EI-05) are gratuitous attack surface.
- `import_zip` accepts a `src: String` from the renderer without `canonicalize` ([item 33](.audit-notes-code.md)).
- No outbound network today; preserve this invariant — when NF-13 (URL preview) lands, default it OFF and document the privacy implication.

### Recovery and rollback needs
- Backup `.prev` + restore on import failure (EI-03).
- Schema migration framework (EI-04) so a broken v0.2 release can be backed out by rolling the EXE back and the DB downgrade is a no-op.
- Consider an "Open data folder" button in Settings so users can hand-recover if all else fails.

### Logging / diagnostics needs
- No logging anywhere today. For a single-user offline app this is mostly OK, but a Settings → "Diagnostic log" toggle that tee's Rust `tracing::error!`/`tracing::warn!` to `app_data_dir()/keepr.log` (with size cap) would let users include a relevant log when filing bugs. Default OFF for privacy.

## Explicit Non-Goals

Ideas considered and rejected, with reasons.

- **Collaboration / real-time co-edit** — out of scope. Keepr is single-user offline by design. ([keep §3](.audit-notes-keep.md)) Adding sync infrastructure changes the product. Users who need this should use real Keep or Notesnook.
- **Location-based reminders** — Google deprecated them in late 2025; battery-hungry on Android; doesn't fit a desktop app.
- **Folders / hierarchy** — Keep's identity is flat. Trilium / AppFlowy / Obsidian own this space.
- **Outliner / block editing** — anti-Keep paradigm. Logseq owns this space.
- **Database views (Kanban / Grid / Calendar)** — AppFlowy / Notion territory.
- **AI features (RAG / autocomplete / categorization)** — out of scope for v0.x. Real Keep users mostly find these noisy. The local-only / no-telemetry promise is part of the product value.
- **Account creation** — Keepr's instant-open UX is strictly better than the alternatives. Stay anonymous-first forever.
- **Telemetry** — none, ever. Single-user offline + no network = the product promise.
- **Feature paywall** — MIT-licensed; no premium tier; don't fragment.
- **Google Docs sidebar integration** — depends on Docs; out of scope.
- **Wear OS / mobile widgets** — depends on a mobile shell that doesn't exist.
- **Custom user scripts attached to notes (Trilium)** — sandboxing nightmare; never adopt.
- **"Things" auto-categorization in search (Keep's ML)** — requires a model + classifier; not worth the dependency.
