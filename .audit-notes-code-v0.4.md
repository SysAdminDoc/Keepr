# Keepr v0.4 code audit

Scope: every regression / bug / quality gap introduced by v0.2 → v0.4. v0.1 findings (covered by `.audit-notes-code.md`) are assumed fixed. Severity legend: **P0** data loss / crash / security · **P1** visible bug or broken flow · **P2** polish / a11y · **P3** nit.

## 1. Recent regressions in v0.3/v0.4

1. **P0 — `reorder_notes` writes outside `notes`-section scope** (`src-tauri/src/commands.rs:1318-1336`). The Rust command overwrites `notes.position` for **every id passed**, regardless of section. Frontend (`NoteGrid.tsx:73-93`) only passes the currently-visible filtered set, but it includes pinned + others — so dragging in the Archive section while sortMode=custom (if it ever leaks) would clobber active-Notes positions. Worse: `reorder_notes` does NOT clear positions of notes not in the list, so after a single drag every untouched note keeps stale positions and the absolute ordering across the full grid becomes incoherent. Fix: write positions relative to a section-scoped order (e.g. `position = section_start + i`) or maintain a per-section position column.
2. **P0 — `set_reminder` UPSERT bypasses the `reminders.id` primary key** (`commands.rs:999-1009`). `INSERT ... ON CONFLICT(note_id) DO UPDATE` updates the row in place but the renderer's `upsertReminder` (`store.ts:387-394`) keys on `noteId`. So the local `Reminder.id` will silently disagree with the DB after the second `set_reminder` call for the same note — `id` in DB stays the original Uuid, while the client cached object built from the second `set_reminder` reply uses the SELECT row (correct). However the `id` returned via `Reminder.id` for the new set is the old PK, not the freshly-minted `id` param. Document or strip the `id` field from the renderer Reminder shape (it isn't used anywhere except as a map key).
3. **P0 — reminder scheduler marks fired BEFORE notification.show()** (`lib.rs:269-289`, `commands.rs:1100-1107`). `take_due_reminders` UPDATEs `fired_at` inside the same query loop. If `notification.show()` throws (permission denied, OS toast service unavailable, COM error during sleep/resume), the reminder is permanently lost — `fired_at` stays set, the bell badge drops, but the user never saw a toast. Fix: collect the rows first, fire notifications, only mark fired on success. Acceptable degradation: don't mark on `Err(_)` so the next 30 s tick retries.
4. **P0 — Empty-note + reminder = orphan blank note with reminder** (`NoteEditor.tsx:628-642` + `563-585`). `setReminderForNote` calls `ensureExistingId()` which creates an empty note via `api.createNote`. If the user then closes the editor without typing anything, the `close()` empty-note logic at `NoteEditor.tsx:319-325` triggers `deleteNotePermanent` — which cascades to the reminder via FK ON DELETE CASCADE only on `notes.id`. But: the local `reminders` store entry created by `upsertReminder` is **not** cleaned up, leaving a dangling reminder in `useStore` until next `load()`. Worse, the user just set a reminder and the app silently throws away both the note and the reminder. Either suppress the empty-note delete when a reminder/attachment was added, or warn the user.
5. **P1 — Bulk operations race against section-switch effect** (`App.tsx:167-175`). The "clear stale selections" effect runs whenever `notes` changes; bulkTrash patches notes to `trashed:true` which in the Notes section makes them disappear from `filtered`, but they're still in `notes`. They stay selected even though invisible. Then `clearSelection()` runs at end of `runBulk` so this isn't fatal — but during the iteration, intermediate renders show "5 selected" while only 2 cards remain visible. Switch BulkActionBar's count to the visible-and-selected intersection.
6. **P1 — Drag-reorder ID mismatch when section is non-Notes** (`NoteGrid.tsx:73-93`). `useStore` is referenced via `void useStore` (line 145) but `onDragEnd` only reads from the `notes` prop. Inside Archive section the notes list is the *archived* slice, but `api.reorderNotes(positionedIds)` will still overwrite `position` for those IDs. Combined with item #1 this means a Custom-sort drag in Archive will reorder archive cards but corrupt the active-Notes ordering on the next render. Frontend should refuse to enable drag in non-Notes sections (today `NoteGrid` enables it whenever `sortMode === "custom"` regardless of section).
7. **P1 — Dragged note's position is lost when sort flips off Custom** (`store.ts:293-298`). `setSortMode` calls `sortNotes(s.notes, mode)` — fine. But while `mode !== "custom"` the `position` column stops being honored, so re-saving a note (which never touches position in `update_note`) is safe. However, when the user flips back to Custom **after creating new notes**, those new notes have `position = 0` and tie-break by `updated_at`. Sort is stable but new notes slot to the top mixed with old position-0 ones in an unpredictable order. Backfill: assign incrementing position on insert, or run a one-shot normalize when entering Custom mode.
8. **P1 — Tray "Quit" is the only graceful exit; ALT-F4 + close button just hide** (`lib.rs:172-182`). The OS taskbar "Close window" sends `CloseRequested`, which `prevent_close()` + `hide()` traps. Users who don't know about the tray icon will think Keepr exited and won't know it's still running — costing them a tray slot, the global hotkey registration, and 30-second reminder sweeps for the rest of the session. At least document this in the toast on first hide, or expose a Settings toggle: "Close button quits Keepr".
9. **P1 — Global hotkey registration fails silently** (`lib.rs:256-258`). Failure to register `Ctrl+Alt+N` only writes `eprintln!`; no UI surface. User has no way to know why "press Ctrl+Alt+N to open Keepr" doesn't work, and no UI to remap. Add a renderer-side `try-register-on-startup` and emit a toast on failure.
10. **P1 — Auto-backup tick floods after long sleep** (`App.tsx:120-138`, `autoBackup.ts:16-28`). `isBackupDue` returns true once the cadence window elapsed but never **how many** windows; the tick fires `exportZip` exactly once per 30-minute interval. If Keepr was open 13 hours and "daily" was due 12 hours ago, **only one backup runs**, not 26 catch-up exports. That's actually the correct behavior, but the comment in `App.tsx:117` ("missed scheduled backups") is misleading — there's no concept of missed windows, just "is one due right now". Fix the wording or implement a counter.
11. **P1 — `update_note` clobbers attachments-relative `updated_at` in flush race** (`commands.rs:405-419` vs `1241-1245`). `add_image_attachment` bumps `notes.updated_at` to the attach moment; a subsequent `flushDraft()` from the editor — triggered by archive/trash inside `NoteEditor.tsx:500-515` — overwrites it with another `now`. Cosmetic, but two near-identical UTC timestamps stored 30 ms apart are noise; consider squashing the `updated_at` bump in `add_image_attachment` because `update_note` will set it on close anyway.
12. **P1 — Hashtag stays a label after the hashtag text is removed** (`NoteEditor.tsx:272-301`). On every save, `extractHashtagsFromNote` returns tags currently in the text; the code *unions* them into `mergedLabelIds` without removing labels for hashtags that disappeared. So `#work` typed once and then deleted leaves the "work" label permanently attached. Either compute the diff (require the user to remove labels via the chip-X) or document this prominently. (The chip X in `NoteEditor.tsx:829` does work — fine, but discoverable only by users who look.)
13. **P1 — Renaming a label via Edit Labels does NOT prevent re-creation from a remaining hashtag in body** (`NoteEditor.tsx:286-301`). If "work" was renamed to "Workstream" and the note body still says `#work`, next save will call `create_label("work")` and re-attach the recreated label, alongside the rename target. The note now carries both `work` and `Workstream`. Filter the hashtag list against the renamed-out names, or surface a "Update note bodies" prompt when a label is renamed.
14. **P1 — Markdown vault export silently overwrites existing files** (`commands.rs:702`). `std::fs::write(&md_path, content)` clobbers any pre-existing `<title>.md` in the destination folder. No conflict detection across re-exports; if the user re-exports to the same folder after editing notes outside Keepr, those external edits are silently lost. Either error on existing file, prefix names with a timestamp, or write to a fresh subfolder per export run.
15. **P1 — Vault export collision counter caps at 999 and falls back to the **plain** id without uniqueness check** (`commands.rs:649-654`). If `used_names` already contains the 8-char id (unlikely but possible), the loop breaks and a second file with the same name gets written. Either add a final `if used_names.contains(&name)` check before insert, or use the full UUID.
16. **P1 — Takeout import drops `userEditedTimestampUsec` / `createdTimestampUsec`** (`commands.rs:814-865`). Imported notes get the current moment for both `created_at` and `updated_at`, destroying the original chronology. Map Takeout's microsecond fields to RFC3339 and write them via a new `create_note_with_timestamps` path. Already called out in audit prompt; verified true.
17. **P1 — Takeout import drops reminders entirely** (`commands.rs:814-865`). Takeout JSON includes `reminders[]`; the importer never reads it. v0.4 has a reminders table — wire it up or call this out in CHANGELOG/README so users don't expect it.
18. **P2 — Takeout import skips trashed notes — correct, but undocumented in code** (`commands.rs:828-830`). Behavior is right (Takeout-trashed notes are likely already-deleted on Keep's side), but it deserves at minimum an inline `// matches Keep web's behavior` comment so a future contributor doesn't "fix" it.
19. **P1 — Checked-items group still shows drag handle space, no handle** (`NoteEditor.tsx:106` vs `94-107`). `ChecklistRow` renders a `<span className="w-[18px]" />` when `draggable=false`. That's 18px of whitespace per row that looks like a broken drag handle. Either drop the spacer or render a faint dot so the row layout reads intentionally.
20. **P1 — Tray icon falls back to 1×1 transparent ARGB on icon-load failure** (`lib.rs:218-223`). The `default_window_icon()` lookup returns `None` if the bundle config's icon list doesn't include a PNG. `tauri.conf.json:32-38` lists `.png` icons so it should always succeed in release. But `cargo run` from `src-tauri/` without first running `tauri build` produces a build that **does** lack the bundle icon, falling through to the 1×1 transparent — invisible to the user, looks like the tray icon "didn't install". Add a `cfg(debug_assertions)` warning print or embed a PNG via `include_bytes!`.
21. **P1 — `AppState.clone()` in `import_takeout` is `State<'_, AppState>::clone()`, which doesn't exist** (`commands.rs:869, 885, 892`). What's actually called is `tauri::State<'_, T>` copying — `State<T>` is `Copy` because it's a reference. The reads work; but the *intent* the comment implies (sharing the underlying `AppState`) is misleading. No bug, document.
22. **P1 — `export_vault` calls `list_notes(state.clone())` while still holding label-fetch context** (`commands.rs:636`). The label-fetch block at 618-634 takes the mutex and drops it before line 636; `list_notes` re-acquires it. Two lock acquisitions per export — fine for now but holding both atomically inside one transaction would let you guarantee a consistent snapshot. Low priority.
23. **P1 — Escape clears selection even when a modal is open** (`App.tsx:88-90`). The handler runs on every Escape regardless of `editorOpen`/`settingsOpen`/`labelsManagerOpen`. Each modal also has its own Escape handler via `useEscape`. Order of execution: the App-level handler fires *before* the modal handlers register their own listeners (registered later in tree mount order get fired first because they're appended to `document`). User in editor presses Escape → selection clears AND editor closes. Visible only if selection was non-empty and editor was open simultaneously, which is rare but possible. Gate the App handler on `!editorOpen && !settingsOpen && !labelsManagerOpen && !helpOpen`.
24. **P2 — System theme listener never unregisters** (`store.ts:210-221`). Registered once at module load with no removal. Fine in production (lives for app lifetime), but in vitest watch mode the listener accumulates across module reloads. Wrap in HMR-safe registration or document the lifetime.
25. **P2 — Days-left badge in Trash only updates on render** (`NoteCard.tsx:205`, `trashRetention.ts:11-23`). `daysLeftInTrash` is called every render and snapshots `Date.now()`. The card stays mounted past midnight if the user leaves it open, so the badge ticks down only when something triggers a rerender. The hourly sweep in `App.tsx:95-114` does mutate `notes`, but only when something expires. Add an hourly `setState({rev: rev+1})` ping at the App level to force a card refresh, or just live with the freshness.
26. **P2 — "Make a copy" inside the editor closes the editor unexpectedly** (`NoteEditor.tsx:657-670`). After `duplicate`, `closeEditor()` is called unconditionally. Users may expect to continue editing the source. Keep's actual behavior is to close the source and open the copy; we close the source and don't open the copy. At minimum, immediately re-open the copy.
27. **P3 — Tomorrow morning preset bumps to day-after-tomorrow if the user opens it after 8 PM** (`ReminderPicker.tsx:51-56` + `38-44`). `setAt` adds a day when the chosen time is ≤ now. So at 21:00 hitting "Tomorrow morning, 8 AM" should land on tomorrow 8 AM (which is in the future — correct). But "Later today, 6 PM" at 19:00 will roll to **tomorrow 6 PM**, which the label "Later today" no longer reflects. Either disable the preset when stale or update the label.
28. **P3 — `nextWeek` preset is hardcoded to next Monday but labeled "Next week" generically** (`ReminderPicker.tsx:57-63`). Acceptable, but the preset label should say "Next Monday" — which it does (`ReminderPicker.tsx:106`). The function name `nextWeek` is the only drift; rename to `nextMonday`.

## 2. Performance with new code

29. **P1 — `list_notes` attachment query is unbounded** (`commands.rs:276-299`). Reads every attachment row in the DB into memory in a single Vec, then stitches via HashMap. At 1000 notes × 5 attachments = 5000 rows × ~12 columns of String = several hundred KiB of work per load. Fine today, but `Attachment` carries `filename` (up to 255 chars) and full string clones — consider trimming the `filename` from the list_notes response (only `id`, `mime`, `note_id`, `position` matter for the card mosaic), serving the full record from a per-attachment endpoint.
30. **P1 — AttachmentGrid loads originals at preview size** (`AttachmentGrid.tsx:62-68`). `<img loading="lazy" object-cover>` at a 250 px card slot will decode a 2000 px JPEG (~6 MB RAM per image). Five-image note × 8 visible cards = 40 full images in memory; Chromium will tank on a 1000-note grid. Generate a thumbnail blob in `add_image_attachment` (e.g. `image` crate) and serve a `?thumb=1` variant from `keepr-resource://`. P0 for image-heavy users; P1 average.
31. **P2 — Reminder scheduler polls every 30 s — fixed interval** (`lib.rs:265-290`). A reminder set for "now + 1 min" can fire up to 60 s late (worst-case alignment with the prior tick). Acceptable for v1, but use the next-scheduled `fire_at - now` to compute a shorter sleep when the soonest reminder is within 30 s.
32. **P2 — Hashtag parser walks every save** (`NoteEditor.tsx:281-301`). Cheap per-save but unbounded — a 64 KiB body × 1000 checklist items would chew enough CPU to add a perceptible save delay. The 64 KiB cap on body already bounds this, but document the worst case.
33. **P2 — `sortNotes` runs O(n log n) on every `patchNote`** (`store.ts:353-359`). Every color/pin/archive toggle re-sorts the full list. For a 1000-note grid, that's a few-ms cost per click. Skip the sort when the patch doesn't change `pinned`/`position`/`updated_at`/`title`/`created_at`.
34. **P3 — `list_notes` rebuilds the HashMap on every call** (`commands.rs:238-242`). Fine, but if you ever cache an index in `AppState` you save microseconds. Defer.

## 5. Tauri / Rust correctness

65. **P1 — Capability `global-shortcut:allow-is-registered` granted but never invoked** (`capabilities/default.json:11`). Frontend never calls it; only `register` is used from Rust setup. Drop the permission.
66. **P1 — `notification:allow-is-permission-granted` + `allow-request-permission` granted but unused** (`capabilities/default.json:15-16`). Rust calls `.show()` directly via the plugin; the renderer never checks permission status. Drop both.
67. **P1 — Reminder scheduler thread is fire-and-forget — no JoinHandle, no shutdown** (`lib.rs:265-290`). On app exit (`app.exit(0)` from Quit) the thread is killed mid-tick; if mid-`take_due_reminders` it's mid-transaction. WAL recovery handles this but is not free. Use Tauri's `app.run` exit hook or a `tokio::time::interval` task scoped to the runtime so it cancels cleanly.
68. **P1 — `take_due_reminders` is a non-`#[tauri::command]` `pub fn`** (`commands.rs:1055`) — fine, but the `state: &AppState` parameter dodges the `State<'_, _>` wrapper used elsewhere, breaking the convention. Either tag it `#[tauri::command]` and call via `app.invoke_handler` from the scheduler, or document why it's an internal-only helper.
69. **P2 — `import_takeout` writes attachment files **then** inserts rows** (`commands.rs:929-938`). If the row insert fails after the file write, you get an orphan file. Reverse: insert metadata row first, then write file, then DELETE row on write failure (mirroring `add_image_attachment`).
70. **P2 — `validate_zip_archive` rejects oversized backups during `import_zip` but `export_zip` has no analogous caps** (`commands.rs:1569-1631`). A user with > 2 GiB of data in `data_dir` will write a backup they can never restore. Either error at export time or document.
71. **P2 — `export_zip` collects entire file into a `Vec<u8>` before writing to the zip** (`commands.rs:1620-1623`). 500 MiB attachment = 500 MiB RAM spike. Use `std::io::copy(&mut f, &mut zip)`.
72. **P2 — `set_reminder` validates fire_at as RFC3339 but accepts any past time** (`commands.rs:990-993`). Setting a reminder for 2020 fires immediately on next sweep. Reject `fire_at < now - 1 minute` or warn.
73. **P2 — `reorder_notes` runs N UPDATEs in one tx but no `COALESCE(MAX(position))` reservation** — concurrent reorder from a different window (if single-instance ever lapses) would interleave. Single-user app, low risk.
74. **P3 — `commands.rs:1297-1311` delete_attachment doesn't verify the file existed before removing** (`let _ = std::fs::remove_file(path)`). Silent file-missing case is fine, but no telemetry/log.
75. **P3 — `lib.rs:69-71` `Access-Control-Allow-Origin: *` on the custom protocol** is broad — the protocol is renderer-only and Tauri CSP gates the origin, so it's not exploitable, but tighten to `tauri://localhost` if you can.
76. **P3 — `commands.rs:1730-1731` `let _old = std::mem::replace(...)` then `drop(_old)`** — the leading underscore on a `let` binding holds the value live; the explicit `drop` works but is redundant. Use `let _ = std::mem::replace(...)` or just let it fall out of scope.
77. **P3 — No `tauri-plugin-single-instance`** — running `keepr.exe` twice in portable mode opens two windows pointing at the same SQLite WAL DB. SQLite serializes writes via the file lock so corruption is unlikely, but the second instance has its own AppState mutex and the two will fight. Add `tauri-plugin-single-instance`.

## 6. Schema migration completeness

78. **P1 — No backfill of `notes.position` on upgrade** (`db.rs:118-132`). v3 doesn't touch existing tables; users upgrading from v1/v2 have `position = 0` on every note, so the first Custom-sort drag works fine (positions all 0, tie-break stable), but a session that started in Custom mode shows notes in "insertion order, ties broken by updated_at DESC" — which feels random. Add a v4 migration: `UPDATE notes SET position = (SELECT COUNT(*) FROM notes n2 WHERE n2.updated_at > notes.updated_at)` or similar to seed an initial Custom order.
79. **P1 — `attachments` table existed in v2 but had no writers until v4** (`db.rs:100-114`). Documented. No backfill required because the table was empty. Confirm with a v3→v4 migration sanity test: `SELECT COUNT(*) FROM attachments` should not error.
80. **P2 — `reminders.fire_at` is a TEXT column but no `CHECK` constraint** (`db.rs:118-132`) — the Rust validator at `commands.rs:990-993` is the only guard. A direct SQLite write (e.g. by the user opening keepr.db) could insert garbage; the scheduler would then `chrono::parse_from_rfc3339` and error every tick. Add `CHECK (fire_at GLOB '????-??-??T??:??:??*')`.
81. **P2 — `idx_reminders_pending` is partial on `WHERE fired_at IS NULL`** (`db.rs:129-131`) — good, but `take_due_reminders`' query also filters on `snooze_until` and joins `notes.trashed = 0`. The index helps the `WHERE r.fired_at IS NULL` portion only; consider a covering index `(fired_at, fire_at, snooze_until)`.
82. **P3 — `reminders.id` is the PK but `note_id` is `UNIQUE`** (`db.rs:120-121`). The renderer (`store.ts:387-394`) treats `noteId` as the unique key. The `id` column is unused after creation — drop it and make `note_id` the PK.

## 7. UX consistency

83. **P1 — Editor toolbar is crowded — 9 icon buttons + Close** (`NoteEditor.tsx:838-953`). Bell, Image, Color, ListChecks/AlignLeft, Tag, Copy, Archive, Trash, Close = at 250 px modal width, icons overflow. Either group secondary actions under a "More" overflow, or move Copy/Archive into a kebab menu.
84. **P1 — Some `IconBtn` calls pass `pressed`, most don't** (`NoteEditor.tsx:842`, `BulkActionBar.tsx` callsites). Only the Bell button uses `pressed`. Pin/Archive/Color/List-toggle buttons that show a state should all set `pressed` for SR users. Audit every `IconBtn` callsite.
85. **P2 — `aria-pressed` is set on `IconBtn` only when `pressed` is defined** (`IconBtn.tsx:32-34`). When undefined the attribute is `aria-pressed={undefined}` which React drops — correct behavior, but make sure consumers know that omitting `pressed` is intentional. Document.

116. **P1 — Switching Custom → Modified → Custom preserves position correctly** (`store.ts:293-298`, `sortNotes` cmp at 416-435). Verified — `position` survives in `notes[]`, just isn't honored in non-custom modes.

119. **P1 — Empty-note + reminder + close** — already called out at #4.
