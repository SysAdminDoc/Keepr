# Keepr v0.1.0 — Code Audit (HEAD d67484e)

Audit scope: every source file under the repo. Findings are gaps and bugs only — no feature ideas, no positives.
Severity: P0 = data loss / crash / security; P1 = visible bug / broken flow; P2 = polish / a11y; P3 = nit.

---

## 1. Data safety / SQLite

1. **[P0] `import_zip` is vulnerable to zip-slip via absolute paths and `..` traversal** — `src-tauri/src/commands.rs:485-500`. Loop uses `f.enclosed_name()` which protects against `../` segments **only** when the path is relative to the joined base — but the resulting `staging.join(p)` will still happily escape if `enclosed_name` returns a path the joiner reinterprets. Worse, no per-entry size cap, no entry-count cap, no decompressed-size cap (zip-bomb). A malicious backup could fill the disk or write outside `staging`.
   *Fix:* validate `p.components()` for `ParentDir`/`RootDir`/`Prefix` rejection; cap `archive.len()`, individual `f.size()`, and total uncompressed bytes; bail on any entry whose normalized path escapes `staging`.

2. **[P0] `import_zip` is racy across the swap** — `src-tauri/src/commands.rs:511-528`. The "drop-and-replace with throwaway in-memory connection" trick works only if nothing else holds a reference. The frontend can fire another `invoke` in parallel; that command will lock the mutex, get a *valid* throwaway in-memory DB, succeed against it (writing to an in-memory DB that's about to be discarded), then return success while the caller's data was silently lost. There is no global "import in progress" gate.
   *Fix:* add a `busy: AtomicBool` (or a second `Mutex<()>`) the renderer must check, or wrap import in a higher-level "exclusive" lock that rejects writes during swap.

3. **[P0] `import_zip` partial-failure leaves the user with an in-memory DB and no real database** — `src-tauri/src/commands.rs:514-527`. If `std::fs::copy(&restored_db, &target_db)` or `crate::db::open(&target_db)` fails after the throwaway swap, `conn_guard` keeps the throwaway in-memory connection. Subsequent operations succeed against memory and *appear* to work until the app restarts, at which point everything written since the failed import is gone.
   *Fix:* on any error after the throwaway swap, restore the previous file (snapshot to `keepr.db.prev` before copy) and reopen it; never leave the in-memory throwaway as the live conn.

4. **[P0] WAL pages can be lost on export** — `src-tauri/src/commands.rs:444-462`. `export_zip` skips `-wal` and `-shm` (good for portability) but never calls `PRAGMA wal_checkpoint(TRUNCATE)` first. Any committed transactions still in the WAL that haven't been checkpointed are missing from the exported `keepr.db`. The user's most recent writes can silently disappear from the backup.
   *Fix:* before zipping, lock the conn and run `PRAGMA wal_checkpoint(TRUNCATE);`.

5. **[P0] No schema-version table, no migration path** — `src-tauri/src/db.rs:7-48`. `CREATE TABLE IF NOT EXISTS` is the entire migration story. When v0.2 adds an `attachments` table or a column to `notes`, there is no `PRAGMA user_version`, no `schema_migrations` table, no in-code check. A user who installs v0.2 over a v0.1 database will get either silent column-missing errors at query time, or worse, partial schemas applied non-atomically.
   *Fix:* set `PRAGMA user_version = 1` in v0.1 now; add a `migrate(conn)` function that reads the pragma and applies stepped migrations.

6. **[P1] `set_pinned` silently unarchives a note** — `src-tauri/src/commands.rs:302-310`. The SQL sets `archived = 0` whenever pin/unpin happens. There is no UI affordance that pinning from the archive will yank the note out of the archive, and there is no test for this side effect.
   *Fix:* either drop `archived = 0` from the UPDATE, or surface the side effect in the UI (currently the pin button is hidden in trash but visible in archive via the editor — see the editor's archive view, where `setPinned` is reachable through the `draft.pinned` toggle).

7. **[P1] `set_pinned` and `set_color` happily mutate trashed notes** — `src-tauri/src/commands.rs:302-322`. No `WHERE trashed = 0` clause. The renderer hides the buttons in trash so this is currently unreachable from the UI, but the contract is unenforced: a future command or third-party invocation can pin a trashed note. `set_color` on a deleted (already-DELETEd) row is a no-op (returns Ok with 0 affected) — also silently succeeds with no warning.
   *Fix:* either add `AND trashed = 0` to those UPDATEs or check rows-affected and return an error.

8. **[P1] `set_trashed(false)` does not also clear `archived`** — `src-tauri/src/commands.rs:291-296`. Restoring from trash leaves the prior `archived` flag intact, but since notes can be trashed *from* the archive (via the archive UI), restoring them from trash silently sends them back to the active Notes view (because `set_trashed(true)` at line 286-289 already cleared `archived = 0` on trashing). Net effect: archive→trash→restore loses the archive state with no warning to the user. Whether that's intended is undefined.
   *Fix:* spec the desired behavior. If "restore to original section" is wanted, persist the prior section in a column.

9. **[P1] `empty_trash` cascade order is fine for notes/checklist/note_labels but there are no attachments yet** — `src-tauri/src/commands.rs:418-422`. The contract for v0.2 attachments will break: `DELETE FROM notes` cascades the FK rows but does not delete any on-disk files. There is no `BEFORE DELETE` trigger or two-stage delete.
   *Fix:* add a `purge_orphan_attachments` step now, even as a no-op, so the contract is locked in.

10. **[P1] `delete_note_permanent` does not require `trashed = 1`** — `src-tauri/src/commands.rs:253-259`. A future bug or external invocation could hard-delete an active note. No safeguard.
    *Fix:* `WHERE id = ?1 AND trashed = 1`.

11. **[P1] `rename_label` and `create_label` do not enforce the `UNIQUE COLLATE NOCASE` collision the schema declares** — `src-tauri/src/commands.rs:373-386`. `rename_label` will throw a raw rusqlite UNIQUE error string to the renderer with no localization or friendly message. `create_label`'s `ON CONFLICT(name) DO NOTHING` then re-SELECTs by name — if two clients race a creation with the same name, both calls return the same Label, which is OK, but `rename_label` has no `ON CONFLICT` and will fail mid-edit.
    *Fix:* catch UNIQUE violations and return a friendly "label name already exists" string.

12. **[P1] `create_label` returns wrong-case label if the user types a differently-cased duplicate** — `src-tauri/src/commands.rs:344-370`. User has "Work"; user types "WORK". `INSERT … ON CONFLICT DO NOTHING` is a no-op; the SELECT then returns the existing "Work" label, but the renderer doesn't know it was *not* created. This isn't strictly wrong, but UI doesn't show "already exists" feedback.
    *Fix:* compare `id` against the freshly generated UUID; if different, surface "merged into existing label".

13. **[P1] `parking_lot::Mutex` held across `tx.commit()` and a `load_note()` re-read in `create_note` / `update_note`** — `src-tauri/src/commands.rs:194-198, 247-250`. The lock is held for the whole transaction *plus* three SELECTs. This blocks `list_notes`, `set_pinned`, etc. UI auto-saves on backdrop click, so every save fully blocks the next read. With many invocations queued (e.g. user rapid-clicks the masonry), tail latency spikes.
    *Fix:* release the mutex after `tx.commit()` and re-acquire for `load_note`, or split `commands` into a write path that returns the new id and let the renderer call `get_note` separately.

14. **[P2] `notes.position` column is declared but never used** — `src-tauri/src/db.rs:21`, `src-tauri/src/commands.rs:127-134`. `list_notes` sorts by `pinned DESC, updated_at DESC`; nothing writes to `position` for notes. Roadmap v0.2 lists "Drag-and-drop reorder of notes," so a column exists but is dead. Dead columns are usually evidence the schema wasn't migrated forward when the feature was deferred.
    *Fix:* either start writing `position` now (default to created-at epoch ms) or drop the column from v0.1 schema and add it back in v0.2 with a migration.

15. **[P2] `kind` column has no CHECK constraint** — `src-tauri/src/db.rs:14`. Schema comment says `'text' | 'list'` but a typo from a future migration could put `'List'` or `'checklist'` in the column. Same for `color`.
    *Fix:* add `CHECK (kind IN ('text','list'))` and a colors check or move to TEXT-keyed lookup table.

16. **[P2] `created_at` / `updated_at` / `trashed_at` are TEXT** — `src-tauri/src/db.rs:22-24`. RFC3339 strings sort correctly *if* always written with the same timezone offset. `now_iso()` uses `Utc::now().to_rfc3339()` (good), but no constraint enforces that future code paths do the same. A `chrono::Local` accidental swap would silently break sort order.
    *Fix:* add a comment + a unit test asserting all writes go through `now_iso()`. Or store as INTEGER unix ms.

17. **[P2] No FTS index on note title/body/checklist text** — search filtering happens in TypeScript by scanning every note. With 5k notes this is ~5k JS string `toLowerCase().includes(...)` per keystroke.
    *Fix:* `CREATE VIRTUAL TABLE notes_fts USING fts5(title, body, content='notes', content_rowid='rowid');` and route search through SQL.

18. **[P2] No index on `notes.archived` / `notes.trashed` / `notes.pinned`** — `src-tauri/src/db.rs`. All filtering happens after `SELECT id FROM notes ORDER BY pinned DESC, updated_at DESC` returns every row. With many archived/trashed rows the renderer pays N+1 cost on rows it then filters out.
    *Fix:* add `CREATE INDEX idx_notes_state ON notes(archived, trashed, pinned, updated_at DESC);` and let SQL filter (or move filtering into the command).

19. **[P2] `note_labels` primary key `(note_id, label_id)` is fine but missing index on `note_id`** — `src-tauri/src/db.rs:41-46`. The PK covers it as a leading column, so OK; the `idx_note_labels_label` on `(label_id)` is the right index for label-side queries. (No actual issue, but flagging that a future "list notes by label" query at the Rust side has no help.)

20. **[P2] `chrono` 0.4 `to_rfc3339()` includes nanos sometimes** — `src-tauri/src/commands.rs:63-65`. Two consecutive `now_iso()` calls in the same millisecond produce different strings; lexicographic sort still works, but the TEXT column can collect inconsistent precision (`2026-05-25T20:14:00+00:00` vs `2026-05-25T20:14:00.123456789+00:00`) over the project lifetime.
    *Fix:* `Utc::now().format("%Y-%m-%dT%H:%M:%S%.3fZ").to_string()` for stable precision.

21. **[P2] `import_zip` does not remove existing app-data files that aren't in the backup** — `src-tauri/src/commands.rs:502-528`. Code only copies `restored_db` over `target_db`. Any orphan files in `app_data_dir` (future attachments, settings.json, etc.) survive. Contract is undocumented; either it's intentional ("merge") or a bug ("replace"). The Settings modal copy ("Restoring will REPLACE all current notes") implies the latter.
    *Fix:* either delete `data_dir` contents (except `__restore_tmp` and any new sidecar) before copying, or rename the operation to "import" and document the merge semantic.

22. **[P3] `import_zip` `for ext in ["-journal", "-wal", "-shm"]` builds `with_extension("db-journal")` which yields `keepr.db-journal`** — `src-tauri/src/commands.rs:520-521`. `Path::with_extension` *replaces* the extension. `keepr.db.with_extension("db-journal")` is `keepr.db-journal` — actually correct (Path treats `.db` as the ext to swap). Worth a comment or a unit test pinning the behavior; not obvious.

## 2. Tauri 2 / Rust correctness

23. **[P0] `csp: null` allows arbitrary inline script and resource loading** — `src-tauri/tauri.conf.json:25-27`. Tauri's default CSP is a defense-in-depth: with `null` and unscoped fs access (item 24), a single XSS in any note body (rendered as plain text today — but a future markdown render or `dangerouslySetInnerHTML` will change that) becomes RCE-by-file-write through `invoke('write_file', …)`.
    *Fix:* set a real CSP. At minimum `default-src 'self' tauri:; img-src 'self' data: blob: asset:; script-src 'self'; style-src 'self' 'unsafe-inline';`.

24. **[P0] `fs:default + fs:allow-read-file + fs:allow-write-file` give the renderer read/write to anywhere the OS user can reach** — `src-tauri/capabilities/default.json:11-13`. Currently the renderer doesn't invoke the fs plugin at all (export/import go through custom commands that build paths on the Rust side). The plugin is added (`tauri-plugin-fs = "2"`) but unused. Until it's needed, the capability is gratuitous attack surface.
    *Fix:* remove the three `fs:*` permissions and the `tauri-plugin-fs` dependency from `Cargo.toml`. Re-add scoped to a specific path when actually needed.

25. **[P1] `app_data_dir().expect(...)` panics the app on first launch if `APPDATA` is missing/unreadable** — `src-tauri/src/lib.rs:18-22`. Same for `db::open(...).expect(...)`. A user with a misconfigured profile or an admin-redirected `APPDATA` sees a crash with no error window.
    *Fix:* show a Tauri dialog with the error and `std::process::exit(1)` cleanly; the user at least gets feedback.

26. **[P1] `bundle.targets = ["nsis", "msi"]` does not produce a portable EXE** — `src-tauri/tauri.conf.json:31`. README claims "Single portable EXE, ~8 MB" — false. Tauri produces installer wrappers; the raw `keepr.exe` lives at `src-tauri/target/release/keepr.exe` and was never explicitly part of the release artifact list.
    *Fix:* either drop the portable claim, or set `targets = ["app"]` / ship `target/release/keepr.exe` from the GitHub release workflow with a portable-mode data-dir flag.

27. **[P1] No "portable mode" detection** — `src-tauri/src/lib.rs:18-24`. Data dir is always `app_data_dir()` = `%APPDATA%\com.sysadmindoc.keepr` (note: that's the identifier, not "Keepr" — see item 28). For true xcopy portability the EXE must write next to itself when run from a USB stick.
    *Fix:* if `keepr.exe` directory contains a `portable.flag` sentinel file, use `current_exe().parent()` as the data dir.

28. **[P1] `app_data_dir()` resolves to `%APPDATA%\com.sysadmindoc.keepr`, not `%APPDATA%\Keepr` as the README claims** — `src-tauri/tauri.conf.json:5` (identifier) vs `README.md:20`. Tauri 2 uses the bundle identifier for the data dir name by default on Windows; README documents the wrong path.
    *Fix:* either change identifier to `Keepr` (collision risk), or fix README, or override the data dir explicitly.

29. **[P1] `productName: "Keepr"` + bundle `category: "Productivity"` is fine but `longDescription` says "manual ZIP backup" which contradicts roadmap "Auto-backup on schedule"** — `src-tauri/tauri.conf.json:41-42`. Minor doc-vs-future drift.

30. **[P2] No `windows[].center: true`** — `src-tauri/tauri.conf.json:13-23`. First-run window appears in top-left on Windows by default.
    *Fix:* add `"center": true`.

31. **[P2] No `maxWidth` / `maxHeight` / `theme` hint in window config** — `src-tauri/tauri.conf.json`. Tauri 2 supports `"theme": "Dark"` for native title bar matching.

32. **[P2] `#[tauri::command]` functions take `State<'_, AppState>` by value** — `src-tauri/src/commands.rs` throughout. Correct in Tauri 2, but the `.lock()` returns a `MutexGuard` that's held until end of scope. Several commands like `create_note` (line 156-198) hold the guard through `tx.commit()` *and* `load_note()`. See item 13.

33. **[P2] `import_zip` accepts a `src: String` from the renderer and opens it without revalidating** — `src-tauri/src/commands.rs:467-478`. The renderer picks the file via `open()` dialog so it's user-consented, but the path traveled through a JS string with no canonicalization. Symlink-following on Windows is generally fine, but a junction-pointed src can let the operation escape the dialog's "verify" UX intent.
    *Fix:* `std::fs::canonicalize(&src)` and refuse paths under `app_data_dir` to avoid self-overwrite.

34. **[P2] `export_zip` doesn't fsync** — `src-tauri/src/commands.rs:439-464`. `zip.finish()` then `file` drops without `sync_all()`. On Windows a crash within seconds of "Backup saved to…" toast can leave a zero-byte or truncated zip on disk. The user is told it succeeded.
    *Fix:* `let file = zip.finish().map_err(err)?; file.sync_all().map_err(err)?;`.

35. **[P2] `tauri-plugin-dialog` and `tauri-plugin-fs` are added as Cargo deps but only `dialog` is actually used** — `src-tauri/Cargo.toml:17-18`. See item 24.

36. **[P2] `crate-type = ["staticlib", "cdylib", "rlib"]`** — `src-tauri/Cargo.toml:10`. Three crate types for a desktop-only app is overkill (the staticlib/cdylib are for mobile/wasm). Increases build time.

37. **[P2] No `panic = "abort"` profile** — `src-tauri/Cargo.toml`. Release binaries unwind by default; smaller binaries set `panic = "abort"` and `lto = true`.

38. **[P3] `walkdir` crate pulled in only for `export_zip`** — `src-tauri/Cargo.toml:25`. `std::fs::read_dir` recursion would do.

39. **[P3] `serde_json` is in deps but never used** — `src-tauri/Cargo.toml:20`. Dead dep.

40. **[P3] `thiserror` is in deps but never used** — `src-tauri/Cargo.toml:27`. Dead dep; all errors are stringified via the `err()` helper.

41. **[P3] `anyhow` is in deps but only used in `db.rs::open`'s return type** — `src-tauri/Cargo.toml:26`. Could be `rusqlite::Result` directly.

## 3. React frontend bugs

42. **[P0] Note editor close-on-backdrop fires `await api.updateNote` then unmount** — `src/components/NoteEditor.tsx:111-138`. If the user clicks the backdrop, `close()` runs `await api.updateNote(...)` *before* `closeEditor()`. While the await is pending, the user can click backdrop again or press Escape. The `close` handler is re-entrant: two parallel `updateNote` calls will be in flight. Both succeed, the second wins. Worse, if the editor's `existing` ref-changed between the two calls, you get cross-contamination.
    *Fix:* set `busy` state immediately; ignore subsequent close events.

43. **[P0] ALT-F4 / OS window close discards unsaved edits** — `src/components/NoteEditor.tsx`. There is no `tauri.event.listen('tauri://close-requested', …)` to flush in-progress drafts. The `close()` saver runs only on backdrop click / Escape / Close button. Window close kills the renderer; the in-memory `draft` is lost.
    *Fix:* register a `getCurrentWebviewWindow().onCloseRequested(async (e) => { e.preventDefault(); await save(); window.destroy(); })` in `App.tsx` or here.

44. **[P0] `useEffect` reset of draft fires whenever `existing` reference changes** — `src/components/NoteEditor.tsx:67-89`. `existing` is `notes.find(...)` from `useMemo` with deps `[notes, editorNoteId]`. Any other note's update calls `load()` → `set({ notes })` → `notes` is a new array reference → `existing` is a new object reference → `useEffect` reruns → **the user's in-progress edits are clobbered by the latest server state**. Example: user opens note A and starts typing; a separate (debounced/background) refresh re-runs `load()`; mid-typing, the textarea snaps back to the saved server text.
    *Fix:* depend on `[editorOpen, editorNoteId]` only and snapshot `existing` once on open.

45. **[P1] Escape key listener has `[editorOpen, draft, existing]` deps with `// eslint-disable-next-line react-hooks/exhaustive-deps`** — `src/components/NoteEditor.tsx:91-99`. The handler captures `close` via closure; `close` is redefined every render and closes over `draft`. By disabling exhaustive-deps and depending on `[editorOpen, draft, existing]` (without `close`), the effect re-registers a fresh listener on every keystroke (since `draft` changes on every input). This is at best wasteful (add/remove listener per keystroke) and at worst introduces races where two Escape handlers briefly coexist.
    *Fix:* wrap `close` in `useEvent`-style ref or `useCallback` with stable deps; depend only on `[editorOpen]`.

46. **[P1] `NewNoteBar`'s "Take a note…" button is `<button>` containing a left-aligned text but no `type="button"` and not inside a form** — `src/components/NewNoteBar.tsx:10-14`. Default button type is `submit` in forms; here it's outside a form so behavior is OK, but adding the explicit attribute is defensive.

47. **[P1] Multiple NoteCard color pickers can be open simultaneously** — `src/components/NoteCard.tsx:25, 88-200`. `colorOpen` is local state per card; click-outside is not handled. Opening picker on card A then clicking card B's palette opens *both*; clicking elsewhere on the page closes neither. A click that reaches an outer element doesn't close the open picker.
    *Fix:* either lift state to the store (only one open at a time) or add a `useEffect` mousedown listener with `ref.current?.contains(...)`.

48. **[P1] Editor color picker and label menu also lack click-outside dismiss** — `src/components/NoteEditor.tsx:327-337, 348-394`. Open the color picker, click the title textarea, picker stays open and covers the body. Same for the label menu.
    *Fix:* same as item 47.

49. **[P1] `setSection` not exposed in `App.tsx`'s `useStore` destructure, but `App.tsx` re-renders on every store change anyway** — `src/App.tsx:38`. The destructure pulls 7 fields. Any unrelated mutation (e.g. `toast` flipping) triggers `App` re-render → `filterNotes` recomputes (cheap thanks to useMemo) → `<TopBar/>`, `<Sidebar/>`, `<NoteEditor/>` etc. all re-render. With many notes this is fine; with reminders/etc. it'll matter. The store has no selectors-with-shallow-compare pattern.
    *Fix:* use `useStore((s) => s.notes)` with `shallow` from zustand on each subtree.

50. **[P1] `filterNotes` runs on every render; `useMemo` only saves it when `notes`/`section`/`search` are stable** — `src/App.tsx:52-55`. `section` is an object literal returned from `setSection(s)`, and `App.tsx:54` deps include it as a reference. Two `setSection({kind:"notes"})` calls produce different references → `useMemo` invalidates → `filterNotes` re-runs even though logically the section is unchanged. Not a correctness bug, but a perf footgun once you have many notes.
    *Fix:* compare by stable section key, or memoize section in store.

51. **[P1] Search has no debounce** — `src/components/TopBar.tsx:30-36`. Each keystroke writes to store, triggers a `filterNotes` over the entire notes array, and re-renders the masonry grid (which then does DOM measurements). 1000+ notes + fast typist = jank.
    *Fix:* debounce `setSearch` 150ms, or use a separate local input state + commit on blur/250ms idle.

52. **[P1] `NoteCard` `onClick={onCardClick}` fires on every click that bubbles, including from inside the color-picker popover area when `e.stopPropagation` isn't called** — `src/components/NoteCard.tsx:78-87, 157-170`. The picker container has `onClick={(e) => e.stopPropagation()}` (line 161), but the IconBtn buttons inside don't all call `e.stopPropagation` (the IconBtn at line 149 does on the palette via `(e) => { e.stopPropagation(); … }`; archive/trash also do). Inside the ColorPicker, clicking a swatch invokes `onChange(k)` (line 19 in ColorPicker.tsx) which does not stopPropagation, but the outer wrapper in NoteCard at line 161 does — so OK. Verify on touch where the click model differs.
    *Fix:* defensive `e.stopPropagation()` at picker root.

53. **[P1] `NoteCard.tsx:86` `tabIndex={0}` on a `<div>` makes it focusable but with no `role` / no `aria-label` and no Enter/Space key handler** — opening the editor by keyboard is impossible. Pressing Enter on a focused card does nothing.
    *Fix:* add `role="button"`, `aria-label={n.title || "Untitled note"}`, and an `onKeyDown` for Enter/Space.

54. **[P1] `NoteCard` color-picker popover anchors at `top-9 left-0` of `.relative` div** — `src/components/NoteCard.tsx:158-169`. For a card pinned to the right edge of the viewport or near the bottom, the picker can overflow off-screen with no flip logic.
    *Fix:* use a portal + viewport-aware placement, or at minimum mirror to `right-0` based on viewport position.

55. **[P1] Editor `setKind('text')` from a checklist preserves item order but discards `checked` state into nothing** — `src/components/NoteEditor.tsx:150-153`. `body = draft.checklist.map((c) => c.text).join("\n")` drops the `checked` flag. Conversion back to list (line 142-149) does not preserve old `checked` either. Lossy round-trip with no warning.
    *Fix:* either render `[x] foo / [ ] bar` markdown-style in the body, or warn the user.

56. **[P1] `setKind('list')` from text with no body produces a single empty item, but `isEmpty` (line 106-109) still considers `draft.checklist` empty-text as empty** — `src/components/NoteEditor.tsx:140-154`. Convert to list, no entries → close → `isEmpty=true` → if `existing` it's *permanently deleted* via `deleteNotePermanent`. User intended an empty checklist note got destroyed silently.
    *Fix:* don't permanently delete; just `closeEditor` if `isEmpty && existing`. Or treat "kind change with intent" as non-empty.

57. **[P1] `addNewLabel` race when user presses Enter while the previous create is still pending** — `src/components/NoteEditor.tsx:185-192`. Two Enters → two `createLabel` calls → two `load()` calls; the second `setDraft((d) => …)` may run with stale `labels` view that doesn't include the just-added id. The `new Set([...d.labels, lbl.id])` mitigates duplicates but doesn't prevent a stale-state race.
    *Fix:* disable the input while pending; or guard with `isCreatingLabel` flag.

58. **[P1] Editor `archive()` and `trash()` close editor without persisting in-progress draft edits** — `src/components/NoteEditor.tsx:194-208`. User edits title, clicks Trash → only `api.setTrashed` runs; title edits are discarded.
    *Fix:* await `updateNote(...)` first, then `setTrashed`.

59. **[P1] `LabelsManager` and `SettingsModal` close on backdrop click but don't trap focus** — `src/components/LabelsManager.tsx:52-56`, `src/components/SettingsModal.tsx:66-70`. Tab key escapes the modal. Compare with NoteEditor which also doesn't trap focus.
    *Fix:* focus-trap library or a tiny `useFocusTrap` hook.

60. **[P1] `SettingsModal` and `LabelsManager` don't close on Escape** — only NoteEditor handles Escape. Users press Escape; nothing happens.
    *Fix:* shared `useEscape(closeFn)` hook used across all three modals.

61. **[P1] `confirm("Permanently delete all notes in Trash?")` in `App.tsx:42`** — uses browser `window.confirm`. Tauri webview shows the native Chromium confirm (works) but theme is broken vs the dark UI; also the dialog is *modal* which makes screen readers behave oddly. ## 6. Performance

99. **[P0] `list_notes` is N+1 for note row + checklist + labels** — `src-tauri/src/commands.rs:127-147`. For each id from the cheap initial SELECT, `load_note` runs 3 more queries (one for the note, one for checklist, one for labels). With 1000 notes that's 4000+ prepared-statement executions per page load. Every mutation triggers a full `load()` so this happens on every pin/unpin/trash/etc.
    *Fix:* rewrite as 3 bulk queries (`SELECT … FROM notes`, `SELECT … FROM checklist_items WHERE note_id IN (…)`, `SELECT … FROM note_labels WHERE note_id IN (…)`) and stitch in Rust. Or one big LEFT JOIN.

100. **[P1] `load()` always re-fetches both notes and labels** — `src/store.ts:50-53`. Pin a note → reload all notes and all labels. Wasteful.
     *Fix:* mutation-specific reducers.

101. **[P1] Masonry library does layout-thrashing measurement on every children change** — `react-masonry-css` measures DOM dimensions per item per render. With `load()` rebuilding the whole `notes` array, every note unmounts/remounts due to changed array reference. Even though keys preserve identity, the masonry's internal column-balancing recomputes.

102. **[P1] `filterNotes` runs `toLowerCase()` on every note's title + body + every checklist item text on every keystroke during search** — `src/App.tsx:25-33`. No memoization of lowercased strings.

103. **[P2] No virtualization** — at 5000 notes, react renders 5000 NoteCards (with their own state for `colorOpen`) into the DOM. RAM and paint time scale linearly.

104. **[P2] `bgFor` / `borderFor` called on every NoteCard render** — `src/colors.ts:63-70`. Tiny but unmemoized; passes hex strings as inline styles, causing style attribute diffs on every re-render.

## Cross-cutting summary (no fix prescriptions — for the synthesis step)

- Two systemic risks dominate: (1) the import/export pipeline has at least four P0-grade data-loss paths (zip-slip, WAL-pages-lost-on-export, in-memory-throwaway-stuck, no migration plan); (2) the editor has at least two P0 races (`existing`-ref clobber and re-entrant close-handler) that will manifest as silent data loss the first time the user types fast or has two refresh paths overlap.
- The Tauri capability surface (`csp: null` + broad `fs:*`) is gratuitously open given the renderer does not use the fs plugin at all.
- The renderer is roughly one feature flag away from a perf cliff: N+1 in `list_notes`, full-array `load()` on every mutation, no debounce on search, react-masonry-css's measure-every-child behavior, no virtualization.
- Accessibility is essentially absent: no labels, no focus rings, no focus trap, no escape on most modals, no live region for toasts, no keyboard shortcuts. A v0.1 a11y pass is overdue.
- Test coverage is zero. CI does not exist. Three places hold the version string. The "portable EXE" claim is unverified and contradicted by the bundle config.
- Doc drift: README data path is wrong, README portability claim is wrong, ROADMAP marks portable EXE done.
