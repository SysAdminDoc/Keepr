# Keepr Competitive Audit — FOSS Note Apps & Keep Alternatives

Audit date: 2026-05-25. Subject: Keepr v0.1.0 (Tauri 2 + React + Rust + SQLite, portable Windows EXE, ~8 MB, offline-only, single-user, manual ZIP backup).

Goal: extract distribution / storage / sync / backup patterns and concrete UX details from the FOSS note-taking landscape so Keepr can plan v0.2+ without re-inventing or over-scoping. Keepr's identity is the Keep-aesthetic masonry-grid offline card app — anything that pulls it toward outliner / wiki / knowledge-graph territory is out.

---

## 1. anselm94/googlekeepclone — the closest visual reference

**Stack:** React + Material UI + URQL (GraphQL) + Reach Router (frontend); Go + Gorilla Mux + GORM + gqlgen + AuthBoss (backend); SQLite file persistence; Docker multistage + Heroku.

**Distribution:** server-side webapp; Docker image for self-host; live Heroku demo. No native binary, no portable mode.

**Storage:** server SQLite file. No client-side persistence — depends on backend reachability.

**Sync:** real-time WebSocket GraphQL subscriptions across browser tabs / devices for the same logged-in user. Not peer-to-peer.

**Backup/restore UX:** none documented — relies on the operator backing up the SQLite file.

**Notable features Keepr lacks:**
- GraphQL subscription model that pushes note CRUD to every open tab live (cross-tab consistency without polling).
- Account / multi-user mode (AuthBoss cookie sessions) — Keepr is single-user by design, but pattern is here if it ever changes.
- "List view" toggle as a peer to the tile/masonry view.
- `/playground` GraphiQL endpoint exposed to operators in dev.

**UX patterns to copy:**
- Material-UI elevation + Montserrat/Roboto pairing that visually nails Keep — Keepr should sanity-check its own font stack against this reference.
- Color palette is theme-aware (each color has a paired dark-mode variant) — Keepr already does this; this clone confirms it's the right approach.

**Anti-patterns to avoid:**
- A network-required server architecture for what is fundamentally a single-user offline app. Keepr's Tauri+SQLite is the right call; do **not** add a backend just to enable sync (use file-sync instead).

## 2. Joplin — offline-first with optional sync, plugins, E2EE

**Stack:** Electron desktop + React Native mobile + Node CLI. ~3,500 automated tests, mature codebase.

**Distribution:** native installers per OS, mobile app stores, CLI, self-host server (Joplin Server / Joplin Cloud). Web clipper as browser extension.

**Storage:** notes stored in an **open Markdown format** ("retrievable without our application") with a SQLite index per device. Attachments are first-class "resources" stored separately with stable IDs.

**Sync model — 7 first-class backends:**
1. Joplin Cloud (hosted)
2. Nextcloud
3. S3 (any S3-compatible)
4. WebDAV (generic)
5. Dropbox
6. OneDrive
7. **Local filesystem** (this is the one Keepr should care about — it's literally "point at a sync folder" and matches Keepr's "Drop in your Google Drive folder" model)

Drivers expose a filesystem-like interface, so adding a backend is one driver away.

**Conflict model:** when both ends mutated the same note, the local copy is **moved to a "Conflicts" notebook** and the remote replaces local. User then diffs and merges manually. Plugins exist to make this less painful. This is brutal but honest — better than silent last-write-wins.

**Encryption:** opt-in E2EE per profile, applied at the sync layer (data is plain on-device, encrypted before upload).

**Plugin system:** rich plugin API (commands, panels, menus, editors, settings). Plugins are distributed via a community marketplace, signed with a manifest. The plugin-discovery website at `joplin.github.io/website-plugin-discovery` lists them.

**Resource URL scheme:** attachments are referenced in Markdown as `![filename](:/resource-id)` — the `:/` prefix is a Joplin protocol handler that the renderer resolves to the on-disk resource. This is the canonical pattern for "embed an image/file in a note without breaking when the user moves the file." See [https://joplinapp.org/help/apps/attachments/](https://joplinapp.org/help/apps/attachments/).

**Notable features Keepr lacks:**
- Multiple sync backends with one swappable driver layer.
- Attachment-as-resource model (image + arbitrary file embedding).
- WYSIWYG ↔ Markdown dual-mode editor.
- Plugin system (way out of scope for v0.x but worth knowing the precedent).
- Web clipper browser extension.

**UX patterns to copy:**
- **Resource URI scheme** (`keepr-resource://<id>` analog) for embedded images so the path is stable across DB moves, export, and import. Critical if Keepr adds image notes.
- "Local filesystem" sync driver pattern — a single folder Keepr writes to + watches, no opinion about *what* syncs that folder (Drive / OneDrive / Syncthing / Dropbox all work).
- **Honest conflict UX:** when sync detects a conflict, surface both copies; don't silently merge or silently lose.

**Anti-patterns to avoid:**
- The Conflicts notebook is jarring for casual users — better UX is a Keep-style "conflict banner" on the affected card with side-by-side diff in a modal.
- Joplin's plugin marketplace is a security liability — if Keepr ever adds plugins, sandbox aggressively (Tauri's `isolation` pattern).

## 3. Notesnook — trust/security UX

**Stack:** TypeScript monorepo. Web (React) + Desktop (Electron) + Mobile (React Native). NPM (explicitly no yarn/pnpm).

**Distribution:** native apps all platforms, web clipper extension, web app at app.notesnook.com.

**Storage:** local encrypted DB on every device. Crypto: **XChaCha20-Poly1305** for content, **Argon2id** for key derivation. Zero-knowledge — server stores ciphertext only.

**Sync:** custom protocol over hosted Notesnook servers (paid for the official cloud, self-host option exists). End-to-end encrypted by default — not opt-in like Joplin.

**Backup/restore UX:** encrypted backups; import from Evernote / Google Keep / Joplin / Obsidian / HTML / Markdown / plaintext — broad importer matrix.

**Trust UX patterns worth borrowing:**
- **Private Vault** — a separate password-locked partition inside the app for sensitive notes. Even with the app unlocked, vault notes require the vault password to view/edit/delete. Distinct password from app-lock.
- **App Lock** — locks the *entire* database after N minutes of inactivity, regardless of OS lock state. Unlock with PIN, biometric, or hardware security key.
- **Distinction** between the two locks is the magic: app-lock = "anyone with my unlocked laptop can't read my notes"; vault = "even *I* have to prove intent to open these specific notes."
- **Monographs** — publish a single note publicly via an opaque URL (read-only, optionally password-protected, optionally with expiry). Encrypted blob decrypted in the recipient's browser.
- **Reminders** — first-class object attached to notes with recurrence rules.
- **Note version history** — automatic snapshots, browseable, restoreable.

**Notable features Keepr lacks:**
- Vault / app-lock (huge for a single-user offline app — even a simple "PIN to unlock" would be a v0.2 win).
- Reminders.
- Version history.
- Monographs-style public share (out of scope but interesting).
- Web clipper.

**Anti-patterns to avoid:**
- Notesnook's free tier is heavily feature-gated — Keepr is MIT/free and should stay that way.
- The forced-account-creation onboarding is friction; Keepr's "open the app, type a note" path is better — preserve it.

## 4. Memos — single-binary server, microblog timeline

**Stack:** Go (55.9%) + TypeScript (43.4%). ~20 MB single Go binary.

**Distribution:** **single Go binary** for Linux/macOS/Windows; ~20 MB Docker image (one-command deploy); Docker Compose; Kubernetes Helm chart; build-from-source.

**Storage:** SQLite (default), MySQL, or PostgreSQL — DB-agnostic via a driver pattern.

**Sync:** none / not applicable — server-centric model where every client is a thin web UI talking to your one server.

**Backup/restore UX:** depends on backing-up the chosen DB. Recent releases added attachment handling and MCP integration.

**API surface:** **both REST and gRPC** exposed.

**Notable features Keepr lacks:**
- Microblog/timeline ordering (descending chronological by default) — different paradigm, but worth knowing about.
- `#hashtag` inline parsing (every `#word` becomes a tag automatically; tag panel built from extraction). This is way better UX than Keepr's "add a label via menu" — see "5 patterns to adopt" §3.
- Public share-link per memo.
- gRPC API for automation.
- Daily-review email/digest.

**UX patterns to copy:**
- **Inline `#hashtag` tag creation.** Don't make users open a menu to add a label — let them just type `#groceries` in the note body and have the renderer extract & link it. Keep the label menu for assigning *colors* / renaming, but creation should be inline-first.
- **Single-binary distribution.** Keepr's portable EXE is already in this spirit; lean further into it. Don't add a separate installer SKU unless asked — the portable model is the differentiator.
- 0.27.0 changelog notes emoji-variation-selector handling in tag parser — implementation gotcha to remember if hashtags are added.

**Anti-patterns to avoid:**
- Memos' append-only timeline is wrong for Keep's "pin + masonry" paradigm. Don't copy the chronology, just the inline-tag input UX.

## 5. AppFlowy — Notion-class workspace

**Stack:** Flutter (73.8% Dart) + Rust (24.1%). AGPLv3.

**Distribution:** desktop GitHub releases; FlatHub; Snapcraft; SourceForge; iOS App Store; Play Store; self-host option.

**Storage:** offline-first local DB; cloud sync via AppFlowy Cloud (self-host) or hosted. Specific format not disclosed in README but the architecture description and the Rust core strongly suggest a CRDT layer (collab-rs / Yjs-like).

**Sync:** AppFlowy Cloud self-hostable; cloud option also available; data-privacy-first marketing.

**Backup/restore UX:** not detailed in README.

**Notable features Keepr lacks (and shouldn't necessarily want):**
- Grid / Kanban / Calendar database views — way out of Keep paradigm.
- AI assistant / autocomplete (provider unspecified in README — likely OpenAI-compatible).
- Multi-user real-time collab via CRDTs.

**UX patterns to copy:**
- Self-hostable sync server as a *separate* SKU from the app itself — keeps the app installable without "create account" friction. Keepr's local-only mode should remain the default forever.

**Anti-patterns to avoid:**
- AppFlowy's full Notion-clone scope. Resist any pull toward database views, AI block widgets, or block-based editing — those make it stop feeling like Keep.

## 6. Blinko — Tauri-based AI note app

**Stack:** TypeScript (92%) + Rust (2.9%) + CSS (2.3%). Tauri.

**Distribution:** macOS / Windows / Android / Linux via Tauri bundles; Docker for the self-hosted server; PikaPods integration (revenue-share hosted option).

**Storage:** PostgreSQL backend; all data in self-hosted environment.

**Sync:** server-centric (PostgreSQL is the source of truth, client is a thin Tauri shell).

**AI/RAG:** **RAG over personal notes** for natural-language search. OpenAI integration referenced in repo topic tags.

**Notable Tauri packaging lessons:**
- Explicit macOS Gatekeeper guidance in the README for the unsigned-app quarantine attribute — Keepr should add similar docs (`xattr -d com.apple.quarantine Keepr.app`) the moment a macOS build ships.
- Bash one-liner installer for the Docker server — frictionless. Keepr doesn't need this (no server), but the principle of "one paste line to running" applies to the portable EXE story too: just `winget install Keepr` or `irm <url> | iex` would help.

**Anti-patterns to avoid:**
- PostgreSQL backend is overkill for single-user. Keepr's SQLite + Tauri choice is correct; don't mirror this.
- The Tauri 92% TS / 2.9% Rust ratio shows Blinko basically writes everything in the WebView; Keepr should keep more business logic in Rust for performance and to justify the Tauri choice.

## 7. Trilium Notes — power-user hierarchical wiki

**Stack:** TypeScript (51.2%) + Electron desktop + Node sync server.

**Distribution:** Electron desktop install; Docker for the self-hosted sync server; web interface.

**Storage:** SQLite under the hood (per code references), with notes modelled as a tree where a single note can be cloned into multiple positions (one node, many parents).

**Sync:** self-hosted Trilium Sync Server only — no Dropbox/WebDAV/etc.

**Power features (mostly out of scope but worth cataloging):**
- **Note cloning** — same note in N tree positions.
- **Attributes system** — labels + relations as a typed metadata layer; queryable.
- **Scripting** — frontend & backend JS scripts attached to notes (huge security surface).
- **Per-note encryption** (subtree-level).
- **Seamless versioning.**

**Notable features Keepr lacks:**
- Attribute/relation graph between notes (mini Obsidian).
- Note versioning.

**UX patterns to copy:**
- **Per-note encryption** option (vs Notesnook's whole-vault approach) — gives users granularity without forcing them to manage two passwords on day one.

**Anti-patterns to avoid:**
- The hierarchical-tree model is the antithesis of Keep's "every note is a peer on the corkboard." Do not add folders.
- User-scripts-attached-to-notes is a sandboxing nightmare — never adopt.

## 8. Standard Notes — extension architecture + longevity ethos

**Stack:** React app monorepo; Electron desktop; React Native mobile.

**Distribution:** web (app.standardnotes.com), Mac, Windows, Linux, iOS, Android. Self-host option via static files + `DEFAULT_SYNC_SERVER` env var.

**Storage:** local encrypted; server stores ciphertext only.

**Sync:** custom encrypted-sync protocol against the official server or a self-hosted one. Multi-device unlimited on free tier.

**Extension architecture (worth understanding):**
- Extensions load in a **sandboxed `<iframe>`** inside the app shell. Devs can use any JS framework inside.
- Communication is a postMessage component-API contract.
- Premium ("Plus") editors deliver rich-text, code, spreadsheet, etc. — all gated to paid tier.
- Extensions repo at `github.com/sn-extensions` (archived; ecosystem cooled).

**Longevity guarantee:** product philosophy is "we want to be here in 100 years." Minimal feature creep, paid upgrades fund maintenance, plain-text-first format makes data extractable forever.

**Notable features Keepr lacks:**
- Extension system (sandboxed `<iframe>` model is the most secure approach known).
- E2EE sync server.

**UX patterns to copy:**
- **Iframe-sandbox extension model** if Keepr ever does plugins — it's the safest pattern in the industry.
- **Longevity messaging** in the README — Keepr's SQLite + ZIP-export combo is *already* a longevity story; document it explicitly so users know their notes survive Keepr's death.

**Anti-patterns to avoid:**
- Feature paywall (Plus editors). Keepr is MIT; don't fragment.

## 9. Obsidian — folder-of-markdown + plugin ecosystem (closed source, but influential)

**Stack:** Electron desktop + mobile native shells. Closed source.

**Distribution:** native installers per OS; mobile app stores; portable mode supported on desktop.

**Storage model:** the **vault** = a folder of Markdown files on disk. Attachments live in the same folder (config controls which subfolder). Properties (YAML frontmatter) carry metadata. This is the model that won the local-first note war.

**Sync:** none built-in for free. Obsidian Sync is paid; users widely use iCloud Drive / Syncthing / Git on the vault folder.

**Core/built-in plugins** (~30 — disable-able):
- **Daily Notes** — opens/creates today's note with template variables.
- **Graph view** — note-link graph viz.
- **Canvas** — `.canvas` file = infinite whiteboard of cards / embeds / web pages.
- **Properties** — YAML frontmatter editor with typed fields.
- **Templates** — insert templated content with date tokens.
- **Web viewer**, command palette, slash commands, outline pane, bookmarks, search, file recovery, etc.

**Notable features Keepr lacks:**
- Plain-file-on-disk storage (Keepr is SQLite-only — strong tradeoff discussion needed before changing).
- Daily-note pattern (could fit Keep as "create today's checklist" template).
- Graph view (out of scope).
- Properties / typed metadata.
- Massive third-party plugin marketplace.

**UX patterns to copy:**
- **Vault-as-folder concept** for export: even if Keepr keeps SQLite internally, its ZIP export should also offer a "Markdown vault" export mode (one `.md` per note, attachments in `_resources/`, labels in YAML frontmatter). This is a longevity insurance policy that costs ~half a day to implement.
- **Portable mode** on desktop — Keepr already does this; Obsidian validates the demand.

**Anti-patterns to avoid:**
- Closed source. Don't.
- Obsidian's `app.json` / `appearance.json` / `core-plugins.json` config sprawl — Keepr's settings should stay in one DB row or one TOML, not a config-folder explosion.

## 10. Logseq — outliner + daily journals + block-graph

**Stack:** ClojureScript + Electron + mobile shells. AGPLv3.

**Distribution:** Electron desktop + mobile.

**Storage:** **each day = one Markdown (or Org) file** in `journals/`; non-journal pages live in `pages/`. Block-bullet structure is preserved as indented Markdown lists. Newer "Database version" with Real-Time Collaboration is in alpha.

**Sync:** file-based; works with any folder syncer. Paid Logseq Sync exists.

**Notable features Keepr lacks (mostly out of scope):**
- Block-level addressing (`((block-uuid))` references).
- Outliner editing model.
- Backlinks panel.
- Graph view.
- Plugin marketplace.

**UX patterns to copy:**
- **Daily journal**: if Keepr ever adds a "today's note" quick-action (Cmd+T → opens today's checklist), the Logseq pattern is the reference. Filename = ISO date; template configurable.

**Anti-patterns to avoid:**
- The outliner paradigm replaces flat cards with nested bullets — pure poison for Keep's identity. Skip entirely.

## 12. Built-in OS sticky notes

### Windows 11 Sticky Notes (the new OneNote-backed version)

- Drag the note window to reposition on the desktop; resize by edges/corners; position remembered across launches.
- Microsoft-account sync across devices and to a web view.
- Right-click taskbar icon → "Show all notes" / "Hide all notes" — global toggle for desk-clutter.
- Right-click a note → Color (preset swatches), "Pin to Top" (always-on-top window-level).
- Color changes sync across devices for signed-in users.

### macOS Notes

- Folder hierarchy + Tags + Smart Folders (saved tag queries).
- Pin notes within a folder.
- Lock note with password / Face ID / Touch ID (but: can't lock a tagged note, can't lock notes in a shared folder — friction users complain about).
- Real-time multi-user share on a note or folder.
- Drag a note between folders.

### KDE Marknote

- WYSIWYG markdown editor.
- Notes saved as **`.md` files in `~/Documents/Marknote/` by default** (configurable folder).
- Notebooks = subfolders, each can have an **icon + accent color**.
- Collapsible sidebar; focus mode hides everything but text.
- Sort by alphabetical or date.

### GNOME Notes (Bijiben — note: dormant)

- Minimal, color-tagged note cards.
- File-based local storage.
- (Project has been deprioritized; not a strong reference anymore.)

**UX patterns to copy:**
- **Windows 11**: global "Show all / Hide all" toggle from the system tray icon. Keepr already lives in `%APPDATA%`; adding a system-tray icon with this toggle is small and feels native.
- **Windows 11**: "Pin note to top" as a window-level always-on-top mode — if Keepr ever pops out individual notes into their own windows, this is the gesture users expect.
- **macOS**: drag-into-folder gesture for moves. Keepr's labels are flat (not folders), so the analog is "drag card onto label chip in sidebar to assign that label."
- **Marknote**: store-as-Markdown-files mode as an option (loops back to Joplin/Obsidian — see "5 patterns to adopt" §1).

**Anti-patterns to avoid:**
- macOS Notes' "can't lock a tagged note" and "can't lock notes in a shared folder" — restrictions that surprise users. If Keepr adds locks, locks must compose orthogonally with labels and trash/archive.

## Cross-cutting themes

### Sync/backup patterns that appear in multiple apps

1. **Local filesystem driver = let an external syncer do the work.** Joplin, Obsidian, Logseq, Marknote all rely on the user's existing Drive/iCloud/Syncthing/OneDrive folder. Keepr is already aligned via "manual ZIP backup to your Drive folder" — the next step is **continuous file sync** (write a folder of files that's safe to sync continuously, see §1 of patterns to adopt) rather than a built-in Keepr sync server.
2. **E2EE-at-rest for sync.** Joplin (opt-in), Notesnook (default), Standard Notes (default) all encrypt before upload. If Keepr ever syncs over user-chosen cloud, it should encrypt before write.
3. **Conflict notebook / banner.** Joplin's "Conflicts notebook" is the universal pattern; the more humane variant is per-card "conflict banner with diff."
4. **Open / extractable format.** Joplin (Markdown), Obsidian (Markdown), Marknote (Markdown), Standard Notes (plain JSON) — all promise data survives the app. Keepr's SQLite + ZIP export is good; a Markdown-vault export mode would seal it.

### Table-stakes features for offline note apps in 2026

Across the field, these show up almost universally; Keepr's status in brackets:

- Pinned + unpinned grid [Keepr: yes]
- Labels/tags [Keepr: yes]
- Search across title + body [Keepr: yes]
- Archive + Trash with restore [Keepr: yes]
- Light + dark theme [Keepr: yes]
- Color-coded notes [Keepr: yes]
- Markdown formatting (at least basic) [Keepr: no]
- Image / attachment support [Keepr: no]
- Inline `#hashtag` tag input [Keepr: no]
- Drag-reorder (checklist items and cards) [Keepr: no]
- Modal/lightbox edit when card is clicked [Keepr: partial — verify]
- Some form of sync, even if it's "write to a folder" [Keepr: manual ZIP only]
- Reminders [Keepr: no]
- Some form of app lock or vault [Keepr: no]
- Import from other apps (at least one) [Keepr: no]
- Export to plain text / Markdown [Keepr: ZIP of internal format only]

### What's "Keep-specific" Keepr should keep doing well

- **Masonry grid as primary view** (not list, not outline, not infinite canvas).
- **Pinned section visually separated** above the rest.
- **12-color theme-aware palette** — the exact Keep swatches.
- **Cards are atomic** — no folders, no hierarchy, no nesting.
- **Single-keystroke create** — typing immediately makes a note, no modal-first workflow.
- **Checklist as a peer to text**, not a block-inside-a-doc.
- **Labels as flat tags**, no namespacing or folders.
- **Search across everything by default.**

### What Keep users would miss most if forced onto Keepr today

1. **Images and drawings.** Keep is heavily used for photo-of-receipt, photo-of-whiteboard, sketch-with-finger. Keepr has none of that. Highest priority gap.
2. **Reminders.** Keep is partly a reminder app; people set "remind me Friday 9 AM" on shopping lists. Keepr has nothing.
3. **Collaborators.** Keep's family-shopping-list use case. Keepr is single-user offline by design — out of scope, but worth acknowledging.
4. **Sync across devices.** Keep is "open phone at store, see list edited on laptop." Keepr is laptop-only. ZIP-to-Drive is a poor man's version.
5. **Voice notes + OCR.** Keep transcribes voice memos and OCRs photos of text. Keepr is text-and-checklist only.
6. **Drag-reorder of notes and checklist items.**
7. **Inline `#hashtag` tag creation** (Keep doesn't even do this — but Memos shows it's strictly better UX than menu-based labeling).

### What Keep users would NOT miss

- **Account creation** — Keepr's instant-open UX is strictly better.
- **Network requirement** — when Wi-Fi is down, Keep is useless; Keepr is unaffected.
- **Google AI suggestions sprinkled into the UI** — most users find them noisy.
- **Workspace upsell / Gemini-Spark integration** (Google I/O 2026 push) — Keepr's value prop is the *absence* of this.
- **Telemetry** — Keepr's local-only model implies none; preserve this.

## 5 specific UX patterns / features Keepr should adopt

These are scoped tightly enough for a coding agent to pick up directly.

### 1. **Markdown-vault export mode alongside the existing ZIP export**

**What:** add a second export option that writes a folder of `.md` files (one per note) plus a `_resources/` subfolder for any future attachments. Frontmatter carries the metadata Keepr's SQLite schema holds.

**Why:** longevity insurance — if Keepr disappears, users can open the folder in Obsidian, Joplin, Marknote, or any text editor. Cheap to implement, huge trust signal.

**Shape:**
```yaml
---
id: 0193f4a1-...
created: 2026-05-25T10:23:14Z
updated: 2026-05-25T10:24:01Z
color: yellow
labels: [groceries, weekend]
pinned: true
archived: false
type: checklist        # or "text"
---
- [ ] Milk
- [x] Eggs
- [ ] Bread
```

**Inspiration:** Joplin's open Markdown format ([https://joplinapp.org/help/apps/](https://joplinapp.org/help/apps/)); Obsidian vault layout; Marknote's "save as Markdown in Documents/" default. Pair with a **"Markdown vault import"** that round-trips the same format so users can move data between Keepr and Obsidian.

### 2. **Inline `#hashtag` tag input (Memos-style)**

**What:** parser pass on note body so any `#identifier` token becomes a label-link. New labels are auto-created on first use. Existing label menu still exists for renaming/coloring.

**Why:** removes the "open menu → pick label → confirm" friction. Keep itself doesn't do this; Memos does and users love it. Cheap to implement in the existing parser.

**Shape:** treat `/#[\p{L}\p{N}_-]+/u` in the note body as a tag. On save, diff extracted tags against the note's current labels; insert / remove rows accordingly. Render hashtags as styled chips inline. Watch out for emoji variation selectors (Memos hit this in 0.27.0).

**Inspiration:** Memos [https://usememos.com/features](https://usememos.com/features); macOS Notes' `#tag` Tag Browser.

### 3. **App lock + private vault (two-tier, Notesnook model)**

**What:** two independent locks.
- **App Lock** = PIN / Windows-Hello / biometric required to open Keepr. Auto-locks after N minutes of inactivity (configurable). Locks the entire UI.
- **Private Vault** = a separate password (NOT the app-lock PIN) that gates a per-note "Move to Vault" action. Vault notes show as redacted placeholders in the masonry grid until unlocked; vault auto-locks faster than the app.

**Why:** Keep famously has nothing like this. Users who use Keep for any sensitive info (banking notes, recovery codes, passwords) currently can't. Keepr being offline makes this *more* important, not less — a stolen laptop is a single point of failure.

**Shape:**
- App lock: store an Argon2id hash of the PIN; on launch, gate UI behind unlock dialog. Tauri's `keyring` plugin for "remember on this device for N days" with OS-level credential storage.
- Vault: derive a per-vault key with Argon2id from the vault password; encrypt vault notes' body+title with XChaCha20-Poly1305 (same primitives Notesnook uses). Vault key never persisted — derived on every unlock.

**Inspiration:** Notesnook [https://help.notesnook.com/app-lock](https://help.notesnook.com/app-lock) + [https://help.notesnook.com/lock-notes-with-private-vault](https://help.notesnook.com/lock-notes-with-private-vault).

### 4. **`keepr-resource://<id>` protocol handler for image / attachment embedding (Joplin model)**

**What:** when image / attachment notes land (and they should — top user-missed feature), reference them in note bodies via a Tauri custom protocol `keepr-resource://<uuid>` that resolves to a SQLite-stored blob (or a sidecar file in `%APPDATA%\Keepr\resources\<uuid>`).

**Why:** stable references that survive note renames, DB moves, exports, and imports. Filesystem-path embedding (the obvious naive approach) breaks the moment the user moves the DB or the ZIP is restored on a different machine. Joplin shipped this in v1.x and it's now the gold standard.

**Shape:**
- Register a Tauri custom protocol in `src-tauri/src/main.rs` that resolves `keepr-resource://<id>` to the resource blob.
- Editor inserts `![alt](keepr-resource://<id>)` when an image is pasted / dropped.
- Export bundles `resources/<id>.<ext>` into the ZIP; import re-registers them.
- Markdown-vault export (§1) rewrites `keepr-resource://<id>` to `_resources/<filename>` relative paths so the export is portable to Obsidian/Joplin.

**Inspiration:** Joplin attachments [https://joplinapp.org/help/apps/attachments/](https://joplinapp.org/help/apps/attachments/) — uses `:/resource-id`. Keepr's analog should be a real URL scheme so the WebView can resolve it without IPC roundtrips.

### 5. **System-tray quick-capture + "show/hide all" toggle (Windows Sticky Notes model)**

**What:** add a system-tray icon with three actions.
- Left-click → toggle main window.
- Right-click → "New quick note" (opens a minimal text-only popup, saves on close/Enter).
- Right-click → "Show all" / "Hide all" notes window toggle.

**Why:** the offline-first value prop is wasted if launching Keepr requires Win-key + type "keepr" every time. Tray-resident is how Windows Sticky Notes wins. Quick-capture popup is how Bear/Apple Notes get used during meetings — type, hit Esc, it's saved.

**Shape:**
- Tauri 2's `tray-icon` plugin (already first-party).
- Bind a global hotkey (default `Ctrl+Alt+N`) to quick-capture popup via `globalShortcut` plugin.
- Quick-capture is a borderless 320×240 always-on-top window with a single textarea. Esc / Enter (after typing) saves to default label "Quick capture" and closes.
- Persist tray-enabled + hotkey in Settings.

**Inspiration:** Windows 11 Sticky Notes tray behavior ([https://support.microsoft.com/en-us/topic/get-started-with-sticky-notes](https://support.microsoft.com/en-us/topic/get-started-with-sticky-notes)); Bear's quick-note hotkey; Apple Notes' "quick note" gesture.
